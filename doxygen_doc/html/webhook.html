Webhook de Mercado Pago — Documento técnico y funcional

Endpoint encargado de recibir y procesar las notificaciones automáticas enviadas por Mercado Pago.

Este módulo gestiona las notificaciones HTTP (webhooks) que Mercado Pago envía cuando ocurre un evento relevante en el ciclo de vida de un pago (por ejemplo, aprobado, rechazado o en revisión).

Resumen funcional

  • Qué hace: Recibe las notificaciones de Mercado Pago y actualiza el estado del pago en la base de datos.
  • Para qué sirve: Permite mantener sincronizado el backend con el estado real de las transacciones realizadas.
  • Cómo funciona:
    1. Mercado Pago realiza un POST al endpoint cada vez que se actualiza un pago.
    2. El sistema lee el cuerpo JSON del mensaje y valida que provenga de una notificación válida.
    3. Con el id de la transacción, se consulta la API oficial de Mercado Pago (para obtener más datos).
    4. Se actualiza el estado del pedido en la base de datos según el resultado (ej: aprobado, rechazado, pendiente).
    5. Se registra la notificación en una tabla de logs para auditoría.

Endpoint del Webhook

Ruta: /webhook.php
Método: POST

Cabeceras esperadas:

Content-Type: application/json
User-Agent: MercadoPago Webhook

JSON recibido desde Mercado Pago

Ejemplo de cuerpo recibido (pago aprobado):

{
"id": 123456789,
"live_mode": true,
"type": "payment",
"date_created": "2025-10-08T15:32:05Z",
"user_id": "999999999",
"api_version": "v1",
"data": {
"id": "987654321"
}
}

Respuestas del Webhook

  • ✅ 200 OK → Cuando la notificación fue procesada correctamente.
  • ⚠️ 400 Bad Request → Cuando el cuerpo JSON está vacío o es inválido.
  • ❌ 500 Error interno → Si ocurre una falla al consultar la API o actualizar la base de datos.

Ejemplo de respuesta exitosa:

{
"success": true,
"message": "Notificación procesada con éxito"
}

Ejemplo de error:

{
"success": false,
"message": "Error al actualizar el estado del pago"
}

Registro y trazabilidad

Cada notificación procesada puede almacenarse en una tabla auxiliar (por ejemplo webhook_log) para mantener un historial completo de eventos, facilitando la auditoría y depuración.

Estructura sugerida de la tabla webhook_log:

Campo Tipo Descripción
id INT AUTO_INC Identificador del evento
event_type VARCHAR(50) Tipo de evento recibido (payment, refund...)
payment_id BIGINT ID del pago en Mercado Pago
payload JSON Cuerpo completo de la notificación
fecha_registro DATETIME Fecha de recepción del webhook

Consideraciones de seguridad

  • Validar siempre que el type del evento sea “payment”.
  • Confirmar que el id recibido exista realmente en la base de datos de Mercado Pago (consultando la API).
  • Registrar todos los intentos en logs, incluso los fallidos.
  • Usar HTTPS obligatorio para evitar interceptaciones.

Dependencias

  • db_config.php → conexión segura con la base de datos.
  • m_pago_service.php → módulo encargado de interactuar con la API oficial de Mercado Pago.

Ejemplo de flujo completo

1️⃣ Usuario realiza el pago desde el frontend.
2️⃣ Mercado Pago procesa la transacción.
3️⃣ Una vez completado el proceso, envía un webhook al backend (/webhook.php).
4️⃣ El backend valida y actualiza el estado de la compra.
5️⃣ Si el pago es aprobado, se activan funciones internas (ej: habilitar beneficios Premium).

Nota
Este webhook es idempotente, lo que significa que puede recibir múltiples notificaciones del mismo evento sin causar duplicaciones en la base de datos.
